Organisation d'un projet Low-Code

Structure de projet recommandée

Une bonne organisation est essentielle pour maintenir un projet lisible et évolutif, même en no-code/low-code.

Organisation dans WeWeb

1. Pages et Navigation

  • Pages publiques : Home, About, Contact, Login, Signup
  • Pages authentifiées : Dashboard, Profile, Settings
  • Pages admin : Admin Dashboard, Users Management

2. Organisation des composants

  • Créez des dossiers pour organiser vos composants réutilisables
  • Nommage clair : Card_Product, Button_Primary
  • Documentez l'usage de chaque composant personnalisé

3. Variables et Collections

  • Variables globales : Préfixez avec global_ (ex: global_currentUser)
  • Variables de page : Préfixez avec le nom de page (ex: dashboard_filters)
  • Collections : Noms au pluriel (products, users, orders)

Organisation dans Xano

1. Structure de la base de données

  • Nommage des tables : Singulier ou pluriel cohérent (users, products)
  • Relations claires : Utilisez des foreign keys bien nommées
  • Documentation : Ajoutez des descriptions aux tables et champs

2. Organisation des API endpoints

  • Groupes par ressource : Auth, Users, Products, Orders
  • Nommage cohérent :
    • GET /users - Liste tous les utilisateurs
    • GET /users/{id} - Détails d'un utilisateur
    • POST /users - Créer un utilisateur
    • PATCH /users/{id} - Modifier un utilisateur
    • DELETE /users/{id} - Supprimer un utilisateur

3. Fonctions réutilisables

  • Créez des "Custom Functions" pour la logique répétitive
  • Exemples : send_email, calculate_tax, format_address

Documentation du projet

Ce qu'il faut documenter

  1. Architecture globale : Schéma de l'application
  2. Modèle de données : Schéma de la BDD avec relations
  3. Endpoints API : Liste et description de chaque endpoint
  4. Workflows importants : Processus complexes (paiement, inscription, etc.)
  5. Variables d'environnement : Clés API, URLs, configurations

Outils de documentation

  • Notion : Documentation collaborative
  • Figma : Wireframes et maquettes
  • Miro : Schémas d'architecture
  • README.md : Documentation technique

Gestion de versions et environnements

Environnements recommandés

  1. Development : Tests et développement
  2. Staging : Tests pré-production
  3. Production : Application live

Dans Xano

  • Utilisez les "Branches" pour gérer différentes versions
  • Testez les modifications sur une branche avant de merger

Dans WeWeb

  • Utilisez les "Environments" pour dev/staging/prod
  • Configurez différentes API URLs par environnement

Collaboration en équipe

Bonnes pratiques

  • Rôles clairs : Définissez qui fait quoi (designer, dev, testeur)
  • Conventions communes : Toute l'équipe suit les mêmes règles
  • Revues de code : Même en no-code, validez les workflows complexes
  • Communication : Utilisez Slack/Teams pour coordination

Checklist avant déploiement

  • Tests complets sur tous les parcours utilisateur
  • Vérification de la sécurité (authentification, permissions)
  • Optimisation des performances (cache, pagination)
  • Configuration des variables d'environnement production
  • Backup de la base de données
  • Documentation à jour
  • Plan de rollback en cas de problème
